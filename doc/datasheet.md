# Datasheet: BBO Capstone Dataset

## Motivation

This dataset was created for the Imperial College London Black-Box Optimisation Capstone Challenge. The task requires maximising eight unknown scalar functions using only the input-output pairs returned by an external oracle, with one query per function per week. No analytic form, gradient or internal structure of any function is accessible. The dataset supports the development and evaluation of sequential model-based optimisation strategies under a strict evaluation budget.

The eight functions represent real-world domains including radiation field mapping, noisy ML model scoring, drug discovery, warehouse placement, chemical yield, recipe optimisation, ML hyperparameter tuning and high-dimensional model performance evaluation.

---

## Composition

Each observation is a pair of a D-dimensional input vector in the unit hypercube and a scalar oracle output. The dataset has two components per function: an initial set provided by the course organisers and an oracle history accumulated through weekly submissions.

| Function | Dimensions | Initial N | Oracle queries | Total N | Domain description |
|---|---|---|---|---|---|
| function_1 | 2 | 10 | 13 | 23 | Radiation field, narrow peak, near-zero elsewhere |
| function_2 | 2 | 10 | 13 | 23 | Noisy ML model, stochastic outputs |
| function_3 | 3 | 15 | 13 | 28 | Drug discovery, negative by design |
| function_4 | 4 | 30 | 13 | 43 | Warehouse placement, many local optima |
| function_5 | 4 | 20 | 13 | 33 | Chemical yield, boundary-seeking optimum |
| function_6 | 5 | 20 | 13 | 33 | Recipe scoring, negative by design |
| function_7 | 6 | 30 | 13 | 43 | ML hyperparameter surface |
| function_8 | 8 | 40 | 13 | 53 | High-dimensional ML performance |

Total observations across all functions: 279.

**Data format.** Initial observations are stored as NumPy arrays (`initial_inputs.npy` with shape `(N, D)` and `initial_outputs.npy` with shape `(N,)`). Oracle observations are stored in `oracle_history.json` as paired lists of inputs and outputs per function.

**Known gaps.** After the first several weeks the tight search mechanism concentrated most queries near the current best point for each function. Regions distant from the identified optima are largely unsampled. Function 8 has 53 observations distributed across 8 dimensions, leaving the vast majority of the input space unexplored. The dataset is not a representative sample of any function's domain.

**Stochasticity.** Function 2 is stochastic. The input [0.671, 0.029] was submitted in two separate weeks and returned outputs of 0.417 and 0.385. No other function showed this behaviour in the collected data, though other functions may exhibit noise at finer scales not detected in the available observations.

## Collection Process

The initial dataset for each function was provided by the course organisers as fixed NumPy files and was not generated by the project.

Oracle queries were generated by a Bayesian optimisation solver running on the full accumulated dataset at the time of each submission. The process each week was:

1. Load initial data and all prior oracle responses for the function.
2. Apply any function-specific preprocessing (log transform for function 1, dimension slicing for function 5).
3. Fit a Gaussian Process surrogate with a Matern kernel to all available data.
4. Generate 4096 to 8192 Sobol quasi-random candidates over the search domain.
5. Evaluate the acquisition function (UCB, EI or Thompson Sampling) over all candidates.
6. Refine the acquisition maximum using L-BFGS-B multi-start or Differential Evolution within the search bounds.
7. Submit the resulting point to the oracle and record the returned output.

Whether a function ran in global mode (full [0,1]^D search) or tight mode (local neighbourhood of radius 0.15/sqrt(D) around the current best) was determined by `history.json` state at the time of the run. Tight mode activated when a function showed no improvement for at least two consecutive weeks, subject to per-function threshold overrides in `config.py`.

Data was collected between January and June 2026, one point submitted per function per week for thirteen weeks. All thirteen oracle responses have been recorded per function. The oracle is an external service whose implementation is not accessible.

## Preprocessing and Uses

Two preprocessing transformations were applied before the surrogate was fitted.

**Log transform for function 1.** Raw outputs are near-zero across almost all of [0,1]^2. Fitting a GP directly on these values provides almost no gradient signal. From week 6 onward, all function 1 outputs were transformed with log(y) before fitting. Non-positive values are mapped to min(finite log values) - 10 rather than a fixed floor value, which keeps all training observations on a consistent scale and avoids distorting the GP's normalisation.

**Dimension reduction for function 5.** From week 1 through week 9, the three dimensions x2, x3 and x4 converged consistently to 1.0 at all best-observed points. From week 9 onward, these dimensions were fixed at 1.0 in the dispatcher and the GP surrogate was fitted on x1 alone. The full four-dimensional point is reconstructed before submission. This improves surrogate accuracy for the single free dimension.

**Intended uses.** The dataset is appropriate for evaluating sequential optimisation strategies in the context of this capstone challenge, for studying the effect of different acquisition functions under small budgets and for analysing the bias introduced by adaptive query selection.

**Inappropriate uses.** The dataset is not suitable for training general-purpose regression models, as the query distribution is strongly biased toward regions identified as promising by the optimiser rather than covering the input space uniformly. The total dataset size of 279 points across eight functions is too small for most machine learning applications. Conclusions drawn from these functions should not be generalised to other black-box optimisation tasks without careful consideration of the differences in dimensionality, smoothness and noise level.

## Distribution and Maintenance

The dataset is stored in the project GitHub repository. The initial data files were provided by Imperial College London under the terms of the capstone challenge and are not redistributable. The oracle history and all derived files are maintained by the project author.

There is no automated update pipeline. `oracle_history.json` is updated manually after each week's results are returned by the oracle. The repository is public for the duration of the capstone submission period.

No personal data is contained in the dataset. All inputs are numeric vectors in [0,1]^D and all outputs are scalar real values.
